1. PROJECT OVERVIEW
════════════════════════════════════════

Project Name:    RetailPulse 360
Project Type:    End-to-End Business Intelligence Portfolio Project
Domain:          Indian Retail (FMCG / Multi-city Retail Chain)
Simulation:      12 stores across 5 cities in South India
Data Volume:     646,629 sales rows | 87,840 inventory rows | 1,320 PO rows
Time Period:     January 2022 to December 2024 (3 years)

Business Problem:
  RetailMart India, a mid-sized retail chain with 12 stores across
  5 cities, was losing revenue monthly due to:
  - Inventory blindness (no real-time stock visibility)
  - Supplier delays (no reliability tracking)
  - Siloed reporting (15 different Excel files from 15 departments)
  - No demand forecasting (purchases based on gut feeling)
  - Zero cross-store performance comparison

Solution:
  A centralised, end-to-end Business Intelligence platform that
  connects all data sources into a single star schema data warehouse,
  enabling real-time executive decisions across sales, inventory,
  supply chain, and customer analytics.

2. ARCHITECTURE DECISIONS
════════════════════════════════════════

DECISION 1 — DATABASE: PostgreSQL
Reason:
  - Open source and industry standard
  - Supports complex SQL including window functions and CTEs
  - Handles millions of rows efficiently
  - Native connector available in Power BI
  - Free to use — suitable for portfolio demonstration

Alternative considered: MySQL
Why PostgreSQL chosen: Better support for advanced SQL features
and more commonly used in enterprise data engineering stacks.

----------------------------------------------------------

DECISION 2 — CONNECTION MODE: Import Mode
Reason:
  - Data refreshes daily (not every second)
  - Import Mode stores data in Power BI's in-memory engine
  - Sub-second DAX calculations on 646K rows
  - More suitable for scheduled refresh pattern
  - Better performance for complex DAX measures

Alternative considered: DirectQuery Mode
Why Import Mode chosen: Our use case is daily operational reporting,
not real-time streaming. Import Mode gives 10x better performance
for this scenario.

----------------------------------------------------------

DECISION 3 — DATA MODELING: Star Schema
Reason:
  - Optimized for analytical queries (OLAP workloads)
  - Dimension tables filter fact tables efficiently
  - Power BI's DAX engine is optimized for star schema
  - Easy to understand and extend
  - Industry standard for data warehousing

Alternative considered: Snowflake Schema, Flat Table
Why Star Schema chosen: Best balance between query performance,
model simplicity, and DAX compatibility.

----------------------------------------------------------

DECISION 4 — ETL TOOL: Power Query (M Language)
Reason:
  - Native to Power BI — no additional tools needed
  - Query folding pushes transformations to PostgreSQL
  - Fully auditable — every step recorded
  - Supports complex transformations without code
  - Industry standard for Power BI ETL

Alternative considered: Python ETL, SSIS
Why Power Query chosen: Tightest integration with Power BI,
query folding capability, and zero additional licensing cost.

----------------------------------------------------------

DECISION 5 — DATA SIMULATION: Python (psycopg2 + Faker)
Reason:
  - Real company data is confidential
  - Python allows encoding real business patterns:
    * Diwali/Pongal seasonal spikes
    * Tier 1 vs Tier 2 city demand differences
    * Realistic supplier delay distributions
    * Customer loyalty tier distributions
  - Synthetic data is industry standard for portfolio projects
  - Shows Python skill alongside Power BI skill

Libraries used:
  psycopg2    → PostgreSQL connection and data loading
  Faker       → Realistic Indian names, addresses
  NumPy       → Statistical distributions for realistic data
  random      → Controlled randomization with seed for reproducibility
  datetime    → Date range generation

  - Filter direction: Single (Dimension → Fact)

  Dim_Date     (date_key)     → Fact_Sales (date_key)

  Dim_Store    (store_key)    → Fact_PurchaseOrders (store_key)

Conformed Dimensions:

  to multiple fact tables. This allows cross-fact analysis

11. INTERVIEW TALKING POINTS
════════════════════════════════════════

Q: Why did you choose PostgreSQL?
A: Industry standard open-source RDBMS, supports advanced SQL
   (window functions, CTEs, FILTER clause), native Power BI
   connector, handles millions of rows efficiently.

Q: Why star schema instead of a flat table?
A: Star schema is optimized for OLAP analytical workloads.
   Power BI's Vertipaq engine compresses dimension tables
   extremely efficiently. DAX filter propagation works
   naturally from dimension to fact tables. Easier to extend
   with new dimensions without restructuring fact tables.

Q: Why Import Mode instead of DirectQuery?
A: Our use case is daily operational reporting not real-time
   streaming. Import Mode gives 10x better DAX performance,
   supports all DAX functions, and works better with complex
   time intelligence measures.

Q: How did you handle data quality?
A: Three-layer approach:
   1. Python simulation with controlled randomization (seed=42)
      encoding realistic business patterns
   2. Power Query ETL for null removal, type correction,
      deduplication, and business enrichment
   3. DAX measures using ISBLANK and DIVIDE with
      alternate results to handle edge cases

Q: Explain CALCULATE in one sentence.
A: CALCULATE evaluates an expression in a modified filter
   context — it is the most powerful and most used function
   in DAX because it lets you override the filter context
   that visuals and slicers apply automatically.

Q: What is the difference between SUMX and SUM?
A: SUM aggregates an entire column in one operation.
   SUMX iterates row by row, evaluates an expression for
   each row, then sums the results. SUMX is needed when
   the calculation requires row-level context
   (e.g. quantity × unit_price per row).

Q: How does Row Level Security work?
A: RLS works by applying a DAX filter to a dimension table
   when a user logs in. For Store Manager role I used
   USERPRINCIPALNAME() which returns the logged-in user's
   email and matches it against the manager_name column.
   The filter then propagates through relationships to all
   connected fact tables automatically.
